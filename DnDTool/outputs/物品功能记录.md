# 物品功能记录

## 目标

物品系统需要同时支持规则物品与自定义物品。规则物品来自配置表，自定义物品由玩家在 `ItemInfoEditorUI` 中录入。

`CharacterCreationUI` 不继续承载复杂物品编辑功能，只提供“添加物品”的入口。实际的物品创建、编辑、选择、添加流程统一放在 `ItemInfoEditorUI` 中处理。

## 总体原则

物品不要直接修改角色最终数值。物品提供的数据应作为角色计算的输入，由角色逻辑层统一汇总，最终通过角色详情计算服务生成角色当前状态。

推荐分层：

- UI 层：只负责展示、输入、调用服务。
- 物品服务层：负责创建、编辑、保存、添加物品。
- 角色逻辑层：负责把物品加入角色数据。
- 角色计算层：负责计算物品对角色最终数值的影响。

## 物品数据分层

### 1. 物品基础信息

用于描述物品本体，不直接代表规则效果。

建议包含：

- 物品 ID
- 名称文本 ID
- 描述文本 ID
- 类型
- 子类型
- 稀有度
- 数量
- 重量
- 价格
- 是否可装备
- 是否需要同调
- 是否已经同调

### 2. 结构化效果

用于表达可以自动计算的内容。

例如：

- 属性加值
- AC 加值
- 速度加值
- 先攻加值
- 法术攻击加值
- 法术豁免 DC 加值
- 技能熟练
- 工具熟练
- 武器熟练
- 护甲熟练
- 豁免熟练
- 抗性、免疫、易伤
- 资源上限调整

这些效果应进入角色计算流程，由 `CharacterDetailCalculationService` 等逻辑层服务统一计算。

### 3. 机制标签与参数

部分物品机制可以半结构化表达，不需要一开始全部写成完整自动逻辑。

例如：

- 使用动作
- 附赠动作
- 反应
- 每日次数
- 充能
- 长休恢复
- 短休恢复
- 施放指定法术
- 攻击附加伤害骰
- 指定检定优势
- 指定豁免优势

这类机制可以记录为：

```text
mechanic_type = CastSpell
target = spell_misty_step
value = 1
recovery = long_rest
```

UI 只需要让玩家选择机制类型并填写少量参数，具体是否自动执行可以逐步扩展。

### 4. 自由文本机制

许多 DnD 物品存在非常特殊、低频、需要 DM 判断的机制，不适合强行结构化。

这类内容应允许保存为自由文本，例如：

- 特定剧情条件触发
- 特定生物类型额外效果
- 复杂反应效果
- 需要 DM 裁定的特殊规则

自由文本机制只负责显示在角色卡中，不强制自动化。后续如果某类文本机制频繁出现，再升级为结构化机制。

## 自定义物品录入思路

自定义物品录入建议放在 `ItemInfoEditorUI`。

推荐流程：

1. 玩家从角色创建或其他入口打开 `ItemInfoEditorUI`。
2. 选择“规则物品”或“自定义物品”。
3. 如果是自定义物品，先填写基础信息。
4. 可选填写结构化效果。
5. 可选填写机制标签与参数。
6. 可选填写自由文本机制。
7. 保存后根据入口决定行为：
   - 从 `CharacterCreationUI` 进入：添加到当前角色草稿背包。
   - 从其他界面进入：保存为自定义物品模板，或添加到指定角色。

## 规则物品与自定义物品

建议区分两个概念：

- 物品定义：物品模板，描述一个物品是什么。
- 角色物品实例：角色背包中的具体物品，包含数量、装备状态、同调状态、当前充能等实例数据。

规则物品和自定义物品都可以生成角色物品实例。

## 表格结构建议

当前物品系统如果要支持上述能力，建议中等规模扩展表结构。

建议保留或扩展：

- `TbDndItemDefine`：物品基础定义。

建议新增：

- `TbItemEffect`：物品结构化效果。
- `TbItemMechanic`：物品机制标签与参数。

后续可选新增：

- `TbItemResource`：物品充能、每日次数、恢复规则等资源数据。

## 开发优先级

建议分阶段实现：

### 阶段 1：基础物品与背包显示

- 规则物品可添加到角色。
- 自定义物品可录入基础信息。
- 角色背包能显示物品。

### 阶段 2：静态数值效果

- AC、属性、速度、先攻等静态效果可生效。
- 物品效果进入角色计算服务。

### 阶段 3：熟练项与特性类效果

- 物品可提供技能、工具、武器、护甲、豁免等熟练项。
- 物品可添加显示型特性。

### 阶段 4：资源与充能

- 支持每日次数、充能、短休恢复、长休恢复。
- 角色资源系统统一接入物品资源。

### 阶段 5：复杂机制逐步自动化

- 将高频自由文本机制升级为结构化机制。
- 低频复杂机制继续保留为文本说明。

## 结论

物品系统不应追求一次性自动化所有 DnD 机制。更合理的方向是：

- 常见数值自动化。
- 常见机制半结构化。
- 特殊机制文本兜底。
- 高频机制后续逐步升级。

这样既能快速落地，也能避免 UI 和代码被复杂规则拖垮。
