# DnDTool 完整产品功能架构主线

目标：将 `DM编辑器`、`模块预览`、`角色建卡`、`模块运行游玩` 统一到同一套产品主线下，明确哪些模块是内容生产层，哪些模块是内容验证层，哪些模块是内容消费层，并据此安排后续长期开发顺序。

## 一、产品定位

这套软件不是单一的章节编辑器，而是一套面向 DnD / 跑团场景的“一体化模块工具”。

它至少要同时覆盖四类能力：

1. DM 创建和编辑模块内容。
2. 玩家或 DM 创建和维护角色卡。
3. 作者在正式游玩前预览和验证模块表现。
4. 以运行态方式真正进入模块并进行游玩。

可以把它理解为：

- 一个 `模块制作工具`
- 一个 `角色管理工具`
- 一个 `模块预览工具`
- 一个 `模块运行工具`

这四者不能割裂开发，否则会出现：

- 编辑器数据结构与运行时数据结构不一致
- 预览逻辑与真实游玩逻辑不一致
- 角色卡数据无法进入事件 / 检定 / 战斗系统
- 地图、生物、事件只停留在编辑器层，无法闭环

## 二、总体分层

### 1. 内容生产层

负责“做内容”和“配规则”。

包含：

- `DM编辑器`
- `角色建卡`

### 2. 内容验证层

负责“在正式游玩前验证内容是否正确”。

包含：

- `模块预览`

### 3. 内容消费层

负责“真正运行模块并进行游玩”。

包含：

- `模块运行游玩`

### 4. 核心共享层

负责给上面三层提供统一的数据模型与运行规则。

包含：

- 模块 / 章节 / 地图 / 网格数据
- 事件数据
- 生物模板 / 战场实例数据
- 角色卡数据
- 检定、状态、战斗等基础规则数据
- 持久化与兼容映射

这层是整套产品真正的地基，后续所有模块都应尽量复用它，而不是各自维护一套独立逻辑。

## 三、核心模块拆分

## 1. DM编辑器模块

职责：生产模块内容。

当前已经较明确的子模块：

- `章节编辑`
  - 章节列表
  - 章节基础信息
  - 章节内容组织
- `地图编辑`
  - 地图上传
  - 缩放 / 平移
  - 网格锁定与对齐
- `网格编辑`
  - 格子选择
  - 地形标记
  - 事件绑定区域编辑
- `事件编辑`
  - Trigger / Effect 结构
  - 触发条件编辑
  - 执行效果编辑
- `生物模板编辑`
  - 生物卡录入
  - 预览图
  - 详情面板
- `生物部署编辑`
  - 地图 token 渲染
  - 拖拽部署
  - 选中 / 移动 / 移除 / 激活控制

后续仍需补齐的子模块：

- 实例属性编辑
- 事件与生物实例联动编辑
- 遭遇 / 战斗配置编辑
- 更明确的引用器与选择器 UI

## 2. 模块预览模块

职责：在不正式进入游玩态的前提下，快速验证模块表现。

它不应只是“看静态页面”，而应至少具备以下能力：

- 预览章节切换效果
- 预览地图与网格显示
- 预览事件绑定区域与提示表现
- 预览生物实例显示、隐藏、位置、占格
- 预览基础交互路径是否通顺

设计原则：

- 尽量复用运行态逻辑
- 不单独发明一套“假预览规则”
- 允许 DM 以低成本快速回到编辑态继续修改

它更像是“编辑器内试玩模式”或“轻量运行态”。

## 3. 角色建卡模块

职责：创建和维护玩家角色卡。

建议至少包含：

- `角色基础信息`
  - 名称
  - 种族
  - 职业
  - 背景
  - 阵营
- `角色数值`
  - 六维属性
  - 熟练加值
  - 豁免
  - 技能
- `战斗相关`
  - AC
  - HP
  - 速度
  - 武器 / 法术 / 动作
- `附加信息`
  - 装备
  - 特性
  - 状态
  - 备注

关键要求：

- 角色卡数据必须能被运行态直接消费
- 角色卡应能参与事件检定、剧情判断与战斗系统
- 不应只是一个独立的“表单录入页”

## 4. 模块运行游玩模块

职责：正式加载模块并推进游玩。

这是最终消费编辑器产出内容的地方。

建议分为几层能力：

- `模块加载`
  - 读取模块数据
  - 初始化章节状态
  - 初始化地图、生物、事件
- `游玩主界面`
  - 当前章节展示
  - 地图与角色/生物显示
  - 当前可交互对象
- `事件运行`
  - 触发器判断
  - 效果执行
  - 状态变更
- `角色参与`
  - 检定
  - 状态变化
  - 与地图、生物、剧情的交互
- `战斗运行`
  - 先攻
  - 回合
  - HP / 状态
  - 战斗结束结果

设计原则：

- 运行态是最终真相
- 预览态应尽量靠近运行态
- 编辑器只负责生产运行态需要的数据，不应承载过多运行时专属分支逻辑

## 四、共享数据主线

这部分是后续真正的总主线重点。

## 1. 模块与章节数据

建议作为最外层容器，至少包含：

- 模块基础信息
- 章节列表
- 全局配置
- 角色引用
- 资源引用

章节内继续承载：

- 地图
- 网格
- 事件
- 生物模板
- 生物实例

## 2. 地图与网格数据

职责：

- 提供空间坐标系统
- 承载事件区域
- 承载生物实例部署位置

当前已确认原则：

- 生物实例不写入 `GridCells`
- 地形、事件、单位是三套独立概念
- 生物位置按格子逻辑坐标保存，而不是 UI 像素坐标

## 3. 事件数据

职责：

- 驱动章节交互
- 连接地图、角色、生物、剧情、战斗

当前主线方向：

- 正式收口到 `Trigger / Effect`
- 旧字段只保留最小兼容层

后续角色：

- 事件系统会成为“编辑器配置”与“运行态推进”之间的桥梁

## 4. 生物数据

职责：

- 为地图与战斗提供 NPC / 怪物 / 场景角色

当前已经明确拆分为：

- `生物模板`
- `战场实例`

当前已经明确原则：

- 实例与模板分离
- 实例保存 `InstanceId`
- 实例保存 `IsActive`
- 实例保存格子位置
- 实例允许 DM 单独修改并保存

后续需补齐：

- 实例属性编辑
- 与事件系统的激活 / 移位联动
- 与战斗系统的数据对接

## 5. 角色卡数据

职责：

- 为玩家侧和运行态提供统一角色模型

后续要求：

- 角色卡数据结构要从一开始就考虑进入检定、事件、战斗
- 不要先做成纯 UI 文本表单，再二次硬拆成运行模型

## 6. 规则运行数据

职责：

- 承接真正的“游戏状态”

后续很可能需要独立考虑：

- 角色当前状态
- 生物当前状态
- 章节运行中变量
- 事件已执行状态
- 战斗中状态

这部分应与“静态模块配置数据”区分开。

## 五、主线开发优先级

这里的优先级是“完整产品主线”的优先级，不再只是 DM 编辑器内部优先级。

## P0 共享数据与边界定型

目标：先确定所有模块共用的数据地基。

包含：

- 模块 / 章节顶层结构
- 地图 / 网格结构
- 事件结构
- 生物模板 / 生物实例结构
- 角色卡结构第一版
- 持久化与兼容策略
- 编辑态数据与运行态数据的边界

原则：

- 先定数据职责，再扩 UI
- 先定共享模型，再做各模块消费

## P1 DM编辑器主线

目标：把内容生产层做扎实。

包含：

- 章节编辑
- 地图与网格编辑
- 事件编辑
- 生物模板编辑
- 生物部署编辑
- 基础实例编辑

说明：

- 这是当前进度最靠前的部分
- 但后续开发要始终以“为预览与运行态提供数据”为前提

## P2 模块预览主线

目标：建立第一版内容验证层。

包含：

- 章节流程预览
- 地图与实例预览
- 事件表现预览
- 基础交互预览

原则：

- 优先复用运行态逻辑
- 不要额外发明一套平行规则

## P3 角色建卡主线

目标：建立第一版玩家数据生产层。

包含：

- 角色基础信息
- 角色属性与技能
- 角色战斗字段
- 角色卡持久化
- 角色进入模块运行态的数据接口

说明：

- 角色建卡不是孤立功能
- 它必须提前考虑如何进入检定、事件、战斗

## P4 模块运行游玩主线

目标：让模块真正跑起来。

包含：

- 模块加载
- 章节切换
- 地图交互
- 事件触发与执行
- 角色参与
- 生物联动
- 第一版战斗运行

说明：

- 这是最终消费层
- 也是判断前面数据结构是否合理的最终验收场

## 六、模块依赖关系

为了避免开发时方向跑偏，后续要一直记住下面这条依赖链：

1. `共享数据层` 是所有模块的基础。
2. `DM编辑器` 负责生产模块内容。
3. `角色建卡` 负责生产玩家角色内容。
4. `模块预览` 负责验证模块内容能否正确表现。
5. `模块运行游玩` 负责真实消费前面所有产物。

因此有几条重要原则：

- 编辑器不是终点，只是内容入口。
- 预览不是截图工具，而是运行前验证层。
- 角色卡不是独立页面，而是运行态参与者。
- 运行态不是以后再说的附属功能，而是当前所有结构设计的落点。

## 七、当前阶段判断

截至目前，产品整体所处位置可以这样判断：

- `DM编辑器层`：已进入持续开发阶段，并且已有地图、网格、事件、生物相关基础能力。
- `共享数据层`：已部分成型，但仍需要继续系统化收口。
- `模块预览层`：尚未正式建立为独立主线。
- `角色建卡层`：尚未正式展开。
- `模块运行游玩层`：尚未正式展开。

因此当前最合理的总策略是：

1. 继续推进 DM编辑器，但要始终以共享数据层为中心。
2. 逐步把角色卡、预览、运行态纳入同一份总架构，而不是后补。
3. 后续每做一个编辑器功能，都要问一句：
   - 这个数据将来如何被预览使用？
   - 如何被运行态使用？
   - 如何与角色卡联动？

## 八、当前推荐执行方式

后续建议这样推进：

1. 保留当前 `事件系统主线` 与 `生物部署主线` 作为局部子主线继续开发。
2. 从现在开始，以本文档作为新的“总主线文档”。
3. 后续新增的大功能，都尽量先挂到本文档的模块树下，再进入具体实现。
4. 角色建卡、模块预览、模块运行游玩开始讨论时，也统一回到本文档补充，而不是另起完全脱节的新路线。

## 补充记录：模组运行时框架与权限区分（2026-04-28）

### 运行时定位

模组运行时是正式消费 DM 编辑器产物与角色卡数据的核心系统。预览模式应被视为正式运行时的轻量沙盒版本，而不是一套独立规则。后续不应继续让 `ModulePreviewRuntimeUI` 自己承载大量规则逻辑，而应逐步抽出共享的运行时数据、运行时系统与权限视图层。

核心原则：

- 编辑器数据是配置源数据。
- 运行时数据是会话状态。
- 预览模式使用运行时框架的轻量配置。
- 正式游玩使用运行时框架的完整配置。
- UI 只负责展示与输入，不直接承载复杂规则。

### 运行时功能范围

运行时至少需要覆盖这些能力：

- 模组加载与运行会话管理：从模组配置与角色卡生成 `ModuleRuntimeSessionData`。
- 章节运行：进入章节、切换章节、加载地图、加载生物、加载事件、应用玩家初始位置。
- 地图与单位交互：地图拖动、缩放、网格、地形、玩家 token、生物 token、坐标换算。
- 事件系统运行器：判断 Trigger，执行 Effect，记录事件执行状态。
- 角色参与：角色卡进入运行时后生成玩家角色实例，维护位置、HP、状态、资源等运行态。
- 生物运行状态：在部署实例基础上维护当前位置、当前 HP、状态、是否可见、是否激活、是否进入战斗等。
- 检定与规则结算：能力检定、技能检定、豁免、DC、优势劣势、成功/失败分支。
- 战斗系统：开始战斗、参战单位、先攻、回合顺序、当前回合、结束战斗。
- 运行时存档：保存当前章节、角色状态、生物状态、事件状态、战斗状态、变量状态。

### 推荐框架拆分

- `Runtime Data`：承载运行时状态，例如 `ModuleRuntimeSessionData`、`ChapterRuntimeStateData`、`RuntimePlayerCharacterData`、`RuntimeCreatureInstanceData`、`RuntimeEventStateData`、`RuntimeCombatStateData`。
- `Runtime Builder`：负责从编辑器产物和角色卡构建运行时会话。预览模式和正式运行都应复用它。
- `Runtime Systems`：负责规则与状态变化，不依赖具体 UI。建议拆出 `ChapterRuntimeSystem`、`MapRuntimeSystem`、`EventRuntimeSystem`、`CreatureRuntimeSystem`、`CharacterRuntimeSystem`、`CheckRuntimeSystem`、`CombatRuntimeSystem`、`RuntimeSaveSystem`。
- `Runtime UI`：只负责展示与输入。`ModulePreviewRuntimeUI` 与未来正式 `ModuleRuntimeUI` 都应消费 Runtime Systems。
- `Runtime Event Bus`：负责系统间通知，例如角色移动、事件触发、生物激活、战斗开始、章节切换。

### DM 与玩家权限区分

DM 眼中的软件与玩家眼中的软件必须不同。这个差异不应只做成“隐藏几个按钮”，而应从运行时框架层面加入统一的权限与可见性系统。

建议先定义运行时身份：

```csharp
internal enum RuntimeUserRole
{
    Dm,
    Player,
}
```

DM 视角应拥有完整控制权，包括查看隐藏信息、DM 备注、事件触发区域、隐藏生物、后台变量、调试日志、手动触发事件、调整生物/角色/章节/事件状态、返回编辑器继续修改。

玩家视角只能看到公开信息，包括当前公开地图、自己角色可见信息、已公开文本、已出现的生物、可交互对象、公开事件结果、自己角色状态与可执行操作。玩家不应看到 DM 备注、隐藏生物、事件触发区域、未触发内容、后台变量、调试日志、未公开章节信息。

### 权限框架建议

- `RuntimeAuthority`：决定当前会话身份是谁，以及基础权限是什么。
- `RuntimeVisibility`：决定某个数据对象是否对当前身份可见。
- `RuntimeActionPermission`：决定当前身份是否可以执行某个动作。

长期目标是让同一个运行时状态可以生成不同视图：

- `DMRuntimeViewData`：完整数据、隐藏内容、调试信息。
- `PlayerRuntimeViewData`：仅玩家可见数据。
- `PreviewRuntimeViewData`：面向 DM 的预览数据，并允许切换玩家视角检查暴露内容。

关键原则：

- 运行时状态保存完整数据。
- UI 层不直接读取完整状态。
- UI 层应通过当前身份生成 ViewData。
- 权限过滤应发生在 ViewData 构建层，而不是散落在各个 UI 控件里。

### 后续开发顺序补充

1. 定义 `ModuleRuntimeSessionData` 与 `ChapterRuntimeStateData`。
2. 将预览页现有状态逐步抽象为 Runtime Builder / Runtime State。
3. 定义 `RuntimeUserRole`、`RuntimeAuthority`、`RuntimeVisibility`、`RuntimeActionPermission` 的最小结构。
4. 让预览页先支持 `DM视角 / 玩家视角` 的显示差异。
5. 实现玩家角色运行时实例，支持进入章节后放置到玩家初始位置。
6. 实现事件运行器 MVP，优先支持章节进入、区域进入、生物激活、玩家初始位置、旁白/提示。
7. 再进入检定系统与战斗系统。
