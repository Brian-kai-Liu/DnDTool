# 模组预览运行态主线（第一版）

目标：在点击 `预览` 按钮后，让模组从第一章开始进入“面向 DM 的轻量运行态”，用于验证章节流程、地图表现、事件触发、生物实例显示，并且可以随时退回编辑器继续修改。

## 一、定位

这个“预览”不是正式玩家游玩界面，也不是编辑器界面的简单扩展。

它更准确的定位应该是：

- `面向 DM 的模组预览运行态`
- `基于当前编辑内容生成的运行快照`
- `从第一章开始的模拟运行入口`
- `可随时返回编辑器`

因此它要同时满足两件事：

1. 能让 DM 验证模组当前配置是否能正常运行。
2. 不污染编辑器中的源数据。

## 二、核心原则

### 1. 编辑态与预览态分离

编辑器中的数据是“源数据”，预览中跑起来的数据是“运行快照”。

这意味着：

- 预览中的事件触发不直接改编辑器源数据
- 预览中的生物激活、位置、状态变化不直接回写编辑器源数据
- 预览中的章节推进不直接修改编辑器正在维护的章节配置

否则会出现：

- 预览一次后把正式配置改乱
- 已触发事件被错误保存
- 生物状态、位置、激活结果污染回章节编辑数据

### 2. 第一章作为统一起点

当前建议把“从第一章开始预览”定义为：

- 从当前章节列表顺序中的第一个章节开始
- 即 `index 0`
- 不按章节 ID 排序

这是最符合 DM 编辑心智的方式，因为编辑器里的章节顺序本身就是模组流程顺序。

### 3. 预览是轻量运行态，不是完整正式运行态

第一版预览不追求把所有正式游玩系统全部做完。

它的重点是：

- 验证章节能否开始
- 验证地图和网格表现
- 验证生物实例是否正确显示
- 验证事件是否能按预期触发
- 验证章节切换逻辑是否成立

## 三、第一版预览的功能目标

点击预览后，系统应完成：

1. 先保存当前章节编辑数据。
2. 读取当前模块章节数据。
3. 生成一份运行时快照。
4. 从第一章进入预览界面。
5. 展示当前章节的地图、生物、基础章节信息。
6. 允许 DM 在预览中切换、重置、返回。

## 四、第一版最小功能范围

建议第一版只做这些能力：

### 1. 预览入口

- 从 `ChapterEditorUI` 点击 `预览`
- 先执行保存
- 然后进入预览运行态 UI

### 2. 章节起始加载

- 默认加载第一个章节
- 记录当前预览章节索引

### 3. 地图显示

- 显示章节地图
- 显示地图上的网格
- 保持与编辑器一致的基础对齐逻辑

### 4. 生物实例显示

- 显示当前章节中 `IsActive = true` 的生物实例
- 位置按实例保存的格子坐标渲染
- 占格与体型规则保持一致

### 5. 章节切换控制

- 上一章
- 下一章
- 重新从第一章开始

### 6. 返回编辑器

- 从预览运行态回到 `ChapterEditorUI`
- 默认回到对应章节

## 五、第一版暂不纳入范围

为了避免把预览系统一次做得过重，第一版暂时不要做：

- 完整玩家交互
- 完整角色卡接入
- 完整事件执行器
- 完整战斗系统
- 运行态存档 / 读档
- 多人或玩家视角 UI
- 正式结算与长期状态保留

这些能力都应放到后续“模块运行游玩主线”里逐步建设。

## 六、建议的数据结构层次

预览运行态建议拆成三层：

### 1. 编辑器源数据

来源：

- `ChapterEditorUI` 当前维护的数据
- 或保存后的章节运行时数据

职责：

- 作为预览入口的输入源
- 不在预览过程中被直接修改

### 2. 预览会话数据

建议新增一套“预览会话模型”，用于承载运行快照。

建议目标形态：

```csharp
internal sealed class ModulePreviewSessionData
{
    public int CurrentChapterIndex { get; set; }
    public List<ChapterPreviewRuntimeData> Chapters { get; set; } = new List<ChapterPreviewRuntimeData>();
}

internal sealed class ChapterPreviewRuntimeData
{
    public int ChapterId { get; set; }
    public string ChapterName { get; set; } = string.Empty;
    public string MapImagePath { get; set; } = string.Empty;
    public List<ChapterCreatureInstanceData> CreatureInstances { get; set; } = new List<ChapterCreatureInstanceData>();
    public List<ChapterGridEventData> Events { get; set; } = new List<ChapterGridEventData>();
    public List<ChapterEventBindingData> EventBindings { get; set; } = new List<ChapterEventBindingData>();
}
```

第一版不要求一次把结构完全定死，但至少要满足：

- 当前章节索引
- 章节快照列表
- 章节地图
- 生物实例快照
- 事件与绑定快照

### 3. 预览 UI 状态

预览界面还会有一些不需要持久化的临时状态，例如：

- 当前选中的实例
- 当前高亮格子
- 当前临时面板展开状态
- 当前相机缩放/平移

这部分不应与会话数据混在一起。

## 七、关键流程设计

### 1. 进入预览流程

建议流程如下：

1. 在编辑器点击 `预览`
2. 执行 `SaveChapterEditorState()`
3. 根据当前章节数据构建 `ModulePreviewSessionData`
4. `CurrentChapterIndex = 0`
5. 打开预览 UI
6. 预览 UI 加载第一章内容

### 2. 章节切换流程

建议流程如下：

1. 当前章节索引变化
2. 刷新当前章节地图
3. 刷新当前章节生物实例
4. 刷新章节标题与基础信息
5. 清理上一章的临时选中状态

### 3. 返回编辑流程

建议流程如下：

1. 点击 `返回编辑`
2. 关闭预览 UI
3. 打开 `ChapterEditorUI`
4. 尝试定位回当前预览所在章节

第一版建议规则：

- 返回编辑时不保留预览中的临时运行改动
- 仅保留“回到哪一章”这一层上下文

## 八、DM 预览控制条建议

为了让预览态真正好用，建议第一版顶部就带一条 DM 控制条。

建议包含：

- `返回编辑`
- `重新开始`
- `上一章`
- `下一章`
- `当前章节名`
- `预览模式` 标识

这样 DM 可以快速：

- 来回检查章节衔接
- 对比编辑器配置与运行效果
- 发现问题后立即回去修改

## 九、与当前系统的衔接建议

结合当前项目现状，建议这样落：

### 1. 入口仍从 ChapterEditorUI 发起

当前预览按钮已存在，因此入口不用再讨论。

后续只需将：

- “点击预览”
- “保存”
- “构建预览会话”
- “打开预览界面”

这条链路补完整。

### 2. BattleMainUI 先作为过渡壳

目前项目里已有 `BattleMainUI`。

第一版可以把它作为：

- 预览运行态入口界面
- 或预览系统的临时承载壳

但要明确：

- 它当前名称偏向战斗主界面
- 长期不一定适合作为完整模组预览系统的最终类名

后续可以考虑：

- 直接演进为预览运行态
- 或重命名/拆出更清晰的 `ModulePreviewRuntimeUI`

### 3. 尽量复用现有地图与实例渲染逻辑

当前编辑器侧已经有：

- 地图加载逻辑
- 网格渲染逻辑
- 生物实例 token 渲染逻辑
- 体型占格规则

预览态第一版建议尽量复用这些成熟逻辑，而不是重写一套新的显示流程。

## 十、第一阶段开发顺序建议

建议按下面顺序推进：

### 阶段 A：预览会话入口

目标：

- 让点击预览后，不只是跳界面，而是真正带着第一章快照进入

工作项：

1. 设计 `ModulePreviewSessionData`
2. 从当前章节数据构建快照
3. 将快照传入预览 UI

### 阶段 B：第一章显示

目标：

- 进入预览后能看到第一章的基础内容

工作项：

1. 显示当前章节标题
2. 显示当前章节地图
3. 显示当前章节生物实例

### 阶段 C：预览控制条

目标：

- 让 DM 可以在预览中切换章节并返回编辑

工作项：

1. 返回编辑
2. 上一章 / 下一章
3. 重新开始

### 阶段 D：事件基础预览

目标：

- 验证基础事件是否能在预览态里体现

工作项：

1. 加载事件与绑定
2. 支持最小级别的触发反馈
3. 支持最小级别的实例激活验证

## 十一、当前决策总结

当前关于模组预览运行态，建议先定下这些共识：

1. 预览态从第一章开始。
2. 第一章按当前章节列表顺序中的第一个章节定义。
3. 预览态使用运行快照，不直接污染编辑器源数据。
4. 预览态是面向 DM 的轻量运行态，不是正式玩家运行态。
5. 预览态必须支持随时返回编辑器继续修改。
6. 第一版优先完成章节、地图、生物实例的最小闭环。

## 补充记录：运行时框架与权限视图（2026-04-28）

本节记录“模组运行时需要哪些功能、框架如何搭建、以及 DM/玩家权限视图如何区分”的讨论结果。该记录用于指导后续从 `ModulePreviewRuntimeUI` 逐步抽离共享运行时逻辑。

### 1. 预览与正式运行的关系

预览模式不应是一套独立的假规则。它应是正式运行时的轻量沙盒版本。

后续开发目标是：

- 当前 `ModulePreviewRuntimeUI` 继续作为 DM 验证入口。
- 底层逐步抽出共享运行时会话、运行时系统与权限视图。
- 正式 `ModuleRuntimeUI` 后续复用同一套 Runtime Systems。

### 2. 运行时系统骨架

建议后续新增或逐步形成以下结构：

- `ModuleRuntimeSessionData`：整个运行会话。
- `ChapterRuntimeStateData`：当前章节运行状态。
- `RuntimePlayerCharacterData`：玩家角色运行时实例。
- `RuntimeCreatureInstanceData`：生物运行时实例。
- `RuntimeEventStateData`：事件执行状态。
- `RuntimeCombatStateData`：战斗运行状态。
- `ModuleRuntimeSessionBuilder`：从编辑器数据和角色卡构建运行时会话。
- `ChapterRuntimeSystem`：章节进入、章节切换、章节退出。
- `MapRuntimeSystem`：地图、网格、坐标、单位位置。
- `EventRuntimeSystem`：Trigger 判断与 Effect 执行。
- `CreatureRuntimeSystem`：生物激活、隐藏、移动、状态变化。
- `CharacterRuntimeSystem`：玩家角色位置、状态、资源。
- `CheckRuntimeSystem`：检定结算。
- `CombatRuntimeSystem`：战斗开始、先攻、回合、结束。
- `RuntimeSaveSystem`：正式运行存档与读档。

### 3. 近期优先顺序

近期不建议直接做完整战斗系统，而应先搭运行时骨架：

1. 定义运行时会话与章节运行状态。
2. 将预览页中已有的章节快照、地图状态、生物状态、测试角色状态逐步迁移到运行时会话结构。
3. 定义权限身份与可见性过滤。
4. 在预览页加入 `DM视角 / 玩家视角` 的切换能力。
5. 实现玩家角色运行时实例，先支持初始位置和地图移动。
6. 实现事件运行器 MVP，先支持章节进入、区域进入、生物激活、玩家初始位置、旁白/提示。
7. 再逐步补检定、战斗、正式运行存档。

### 4. 权限视图原则

DM 视角与玩家视角必须从运行时框架层面区分，而不是只在 UI 层临时隐藏按钮。

建议先定义：

```csharp
internal enum RuntimeUserRole
{
    Dm,
    Player,
}
```

并逐步增加：

- `RuntimeAuthority`：判断当前身份与基础权限。
- `RuntimeVisibility`：判断数据是否可见。
- `RuntimeActionPermission`：判断动作是否允许执行。

### 5. DM 可见/可操作内容

DM 应能看到完整运行信息：

- DM 备注。
- 隐藏事件区域。
- 未激活或隐藏生物。
- 后台变量。
- 调试日志。
- 事件执行状态。
- 章节控制。
- 手动触发事件。
- 手动调整角色、生物、章节、事件状态。
- 返回编辑器继续修改。

### 6. 玩家可见/可操作内容

玩家只能看到已公开内容：

- 当前公开地图。
- 自己角色可见信息。
- 已公开文本。
- 已出现的生物。
- 可交互对象。
- 公开事件结果。
- 自己角色状态。
- 当前允许执行的玩家动作。

玩家不应看到：

- DM 备注。
- 隐藏生物。
- 事件触发区域。
- 未触发内容。
- 后台变量。
- 调试日志。
- 未公开章节信息。

### 7. ViewData 原则

运行时状态可以保存完整数据，但 UI 不应直接读取完整状态。

长期建议生成不同视图数据：

- `DMRuntimeViewData`：完整数据、隐藏内容、调试信息。
- `PlayerRuntimeViewData`：玩家可见数据。
- `PreviewRuntimeViewData`：DM 预览数据，可切换玩家视角检查暴露内容。

权限过滤应发生在 ViewData 构建层，而不是散落在各个 UI 控件里。
